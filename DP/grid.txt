62. Unique Paths            A+
INTUITION:-The Dynamic Programming intuition builds the solution by solving for each cell.
        There is only **one** way to reach any cell in the top row (all "right" moves) and **one** way to
        reach any cell in the first column (all "down" moves). For any other cell, the robot *must* have arrived 
        from either the cell directly **above** it or the cell directly to its **left**. Therefore, the total 
        number of unique paths to reach any cell is the **sum** of the paths to the cell above it and the paths 
        to the cell to its left. By calculating this sum for every cell, the value in the bottom-right corner 
        will be the final answer.

63. Unique Paths II          A+
INTUITION:- This problem uses the same Dynamic Programming intuition as the first "Unique Paths" problem, with one 
        simple twist. You still calculate paths to a cell `(i, j)` by adding the paths from the cell **above** (`dp[i-1][j]`)
        and the cell to the **left** (`dp[i][j-1]`). However, if a cell `(i, j)` contains an **obstacle** (`grid[i][j] == 1`),
        you set its path count `dp[i][j]` to **zero**. This $0$ value then correctly propagates, as any path that would have 
        used this obstacle cell is now nullified. This rule also applies to the first row and column: if an obstacle is hit, 
        all cells after it in that line become $0$ because the single path to them is blocked.

64. Minimum Path Sum        A
INTUITION:- This problem is solved using Dynamic Programming by finding the minimum path sum to reach *each* cell. 
            The minimum sum to reach the start cell `(0, 0)` is just its own value. For any cell in the first row, the minimum sum
            is a running total from the left, and for the first column, it's a running total from above. For any other cell `(i, j)`,
            the minimum path to get there *must* come from either the cell **above** `(i-1, j)` or the cell to the **left** `(i, j-1)`.
            Therefore, the minimum sum to reach `(i, j)` is its own value `grid[i][j]` **plus** the **minimum** of the path sums from 
            those two preceding cells. By filling the grid this way, the value in the bottom-right corner will be the overall minimum path sum.

1463. Cherry Pickup II                                       A+ 
* **Simultaneous Movement:** Dono robots hamesha ek sath ek hi row mein aage badhte hain. Isliye do alag rows track karne ke bajaye, state mein sirf ek `row` aur dono robots ke current

576. Out of Boundary Paths                               A+ 
**Intuition / Thought Process:**

* **Core Idea:** Har cell se 4 dishaon (directions) mein explore karna hai aur har step par `remaining_moves` ko ghatana hai.
* **Success Condition:** Jaise hi aapka index grid ke bounds se bahar jata hai, matlab aapko ek successful path mil gaya (`return 1`).
* **Dead End:** Agar aap grid ke andar ho aur `moves == 0` ho jaye, matlab woh rasta aage nahi jayega (`return 0`).
* **Why 3D DP:** Kyunki aap same cell `(r, c)` par same `remaining_moves` ke sath alag-alag raaston se ghoom kar wapas aa sakte hain. Isliye in overlapping 
 states ko dobara calculate karne se bachne ke liye `(row, col, moves_left)` ko memoize karte hain.




576. Out of Boundary Paths                             A+ 
Is approach ka core intuition **"Reverse Boundary Exploration"** hai: hum har possible direction (Up, Down, Left, Right) mein move karke
check karte hain ki kya hum grid se bahar nikal pa rahe hain. Agar koi move grid ki boundary cross karta hai ($r < 0, c < 0$, etc.), toh woh
ek valid "out of boundary" path count hota hai ($+1$ return). Kyunki ek hi cell par baar-baar alag-alag moves ke baad pahunchne ke chances hote
hain, isliye hum **3D Memoization** ($r, c, moves$) ka use karte hain taaki purane calculated results ko reuse kar sakein aur redundancy khatam
ho jaye. Seedhe shabdon mein: har step par 4 raaste hain, aur hum bas wahi raaste ginte hain jo grid ke baahar khatam ho rahe hon.




931. Minimum Falling Path Sum                          A+  
Is problem ka core intuition "Greedy vs Future Impact" hai: hum har cell par khade hokar ye dekhte hain ki agar hum yahan se niche jaana shuru karein, toh agle row ke teen
options (theek niche, diagonal left, ya diagonal right) mein se sabse sasta raasta kaunsa hoga. Recursive approach mein hum Top-Down jaate hain (pehli row se base tak), jabki 
Iterative approach mein hum Bottom-Up (aakhiri row se upar ki taraf) results ko accumulate karte hain. Har cell par decision sirf uske immediate niche wale teen padosiyon ke minimum
value par depend karta hai, jisse ek bada problem chhote sub-problems (har row ka local minimum) mein divide ho jata hai. Revision ke waqt bas ye yaad rakhna: "Har cell ke liye best 
path wahi hai jo uski current value aur uske niche wale best possible move ka sum ho."
